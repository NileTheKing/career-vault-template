# Career Vault — Claude Code Agent Guide

이 vault는 개발자 취업 지원 자료를 관리하는 Obsidian 노트 저장소다.

---

## ⚡ 첫 실행 시 (자동 온보딩)

**세션 시작 시 `projects/` 폴더에 T*.md 파일이 없으면 아래를 자동으로 실행할 것.**

### Step 0. 기존 자료 확인

사용자에게 다음을 물어볼 것:

> "안녕하세요! Career Vault 온보딩을 시작할게요.
> 혹시 기존에 작성한 자료가 있나요? (이력서, 포트폴리오, 자소서, 프로젝트 정리 문서 등)
> **있으면** → `inbox/` 폴더에 넣어주세요. 읽고 원자화해드릴게요.
> **없으면** → 바로 인터뷰 시작할게요. 어떻게 할까요?"

### Step 0-A. inbox/ 파일 있는 경우

1. `inbox/` 폴더의 모든 파일을 읽는다
2. 내용을 분석해 기술 경험 / 비기술 경험으로 분류
3. 각 경험을 `templates/무기_노트_템플릿.md` 포맷으로 원자화
4. `projects/T{번호}_{기술명}.md` 로 저장
5. 비기술 경험은 `me/경험/E{번호}_{경험명}.md` 로 저장
6. `_meta/무기_마스터_목록.md` 인덱스 업데이트
7. 완료 후 생성된 파일 목록과 함께 검토 요청

### Step 0-B. 자료 없는 경우 (인터뷰 모드)

아래 순서로 **한 번에 하나씩** 질문한다. 답변 받으면 바로 파일로 저장하고 다음 질문.

**프로젝트 인터뷰:**
1. "진행한 프로젝트가 있나요? 어떤 프로젝트인지 간단히 설명해주세요."
2. "그 프로젝트에서 어떤 기술적 문제를 해결했나요?"
3. "실측 수치가 있나요? (성능 개선 %, RPS, 에러율 등)"
4. "본인이 직접 구현한 부분은 무엇인가요?"
→ 답변 취합 후 `projects/T{번호}_{기술명}.md` 저장

프로젝트가 더 있으면 반복. 없으면 경험 인터뷰로 넘어감.

**경험 인터뷰:**
1. "인턴, 수상, 학내외활동, 자격증 등 경험이 있나요?"
→ `me/경험/E{번호}_{경험명}.md` 저장

완료 후 `_meta/무기_마스터_목록.md` 인덱스 업데이트.

## Vault 목적

JD가 주어지면 → 관련 무기(경험/수치)를 태그로 빠르게 필터링 → 자소서/포트폴리오 초안 작성

---

## 디렉토리 구조

```
career-vault/
├── inbox/             ← 기존 자료 투하 (이력서, 포폴, 자소서 등) — 온보딩 후 비워도 됨
├── projects/          ← 프로젝트별 원자화 무기 노트 (T1~Tn)
├── me/
│   ├── 경험/          ← 비기술 경험 노트 (E1~En)
│   └── 스킬/          ← 기술 스택별 요약
├── companies/         ← 기업별 JD 분석 + 지원서 (gitignored)
├── templates/         ← 노트 작성 템플릿
└── _meta/             ← 무기_마스터_목록.md, 방법론 문서
```

## 지원서 작성 4단계 프로세스

온보딩 완료 후, JD가 들어오면 아래 단계로 진행. 단계를 건너뛰지 말 것.

- **Step 1**: JD & 회사 심층 조사 → `companies/{회사명}/JD.md` 작성
- **Step 2**: 무기 매핑 — `_meta/무기_마스터_목록.md` 기반 키워드 매핑만 (텍스트 작성 금지)
- **Step 3**: 방향·키워드 확정 — 자소서 문항별 / 포트폴리오 섹션별 키워드·각도 확정 (텍스트 작성 금지, 사용자 확인 후 Step 4)
- **Step 4**: 글 작성 — 방향 확정된 자소서 문항별 텍스트(플레인텍스트) or 포트폴리오 섹션 작성
- **Step 5**: 피드백 루프 — 개발자 면접관 / HR / 서류심사자 입장 각각 시뮬레이션

세부 규칙: `_meta/PORTFOLIO_WRITING_RULES.md`, `_meta/JOB_APPLICATION_METHODOLOGY.md` 참조

## 포트폴리오/자소서 핵심 원칙

- 개조식 bullet 필수 (서술형 금지)
- 역피라미드: 첫 줄에 수치/결과 배치
- 수치 없는 추상 표현 금지 ("크게 향상", "효율적으로" 등)
- 실제 구현 없는 내용 주장 금지

## MCP 툴 가이드 (Smart Connections + Smart Lookup)

MCP 서버(`smart-connections-mcp`) 연결 시 아래 6개 툴 사용 가능.
**경로를 아는 노트는 MCP 대신 직접 파일 읽기** — 빠르고 오버헤드 없음.

둘 다 시맨틱(벡터) 기반이지만 탐색의 두 단계로 사용:
1. `search_notes` (Smart Lookup 역할) → 쿼리로 **처음 노트 찾기**
2. `get_similar_notes` (Smart Connections 역할) → 찾은 노트 기준으로 **연관 무기 확장**

| 툴 | 언제 사용 |
|---|---|
| `search_notes(query)` | 1단계: JD 키워드·개념으로 관련 무기 노트 탐색. 경로 모를 때 진입점 |
| `get_similar_notes(note_path)` | 2단계: 찾은 노트에서 유사 노트 확장 탐색 |
| `get_connection_graph(note_path)` | 여러 경험에 걸친 멀티홉 연결 스토리 엮을 때 |
| `get_note_content(note_path)` | 경로 불확실할 때만. 경로 알면 직접 파일 읽기 사용 |
| `get_stats()` | 임베딩 현황 확인 (몇 개 노트 인덱싱됐는지) |
| `get_embedding_neighbors(vector)` | 고급 사용 — 일반 워크플로우에서는 사용 안 함 |

### Smart Lookup (Obsidian 플러그인)
Obsidian UI 경험용 (1단계 진입점 역할). `search_notes`가 MCP에서 동일 역할 수행.
