# 개발자 취업 지원 방법론

작성자 노트를 정리한 방법론 문서. 자소서 작성 사이클, 글쓰기 요령, 원자화 파일 작성 방식 포함.

---

## 1. 지원서 작성 4단계 사이클

모든 회사 지원 시 반드시 이 순서대로 진행. 단계를 건너뛰면 JD 컨텍스트 없이 쓴 글이 됨.

### Step 1. JD & 회사 심층 조사

**소스 (전부 다 볼 것)**
- 공식 홈페이지 / 제품 페이지 — 서비스 구조, 고객사 규모
- DART — 매출 구조, 최근 실적, 성장세
- 기술 블로그 — 실제 스택, 아키텍처 방향, 엔지니어링 문화
- 보도자료·뉴스 — 최근 기술 투자 방향, 제품 로드맵
- 합격자 후기 (블로그, 커뮤니티) — 실제 면접 질문, 전형 구성

**파악 목표**
- JD 텍스트 너머의 실제 트래픽 규모·특성
- 회사가 실제로 쓰는 기술 스택 (JD에 안 써도 블로그에 나옴)
- 신입/인턴이 실제로 맡는 업무 범위
- 인재상의 실제 맥락 (배경 이유가 있음)

**주의**
- JD만 읽고 바로 글 쓰면 안 됨
- 그룹사 구조 있으면 정확한 법인 확인 (ex. A그룹의 B서비스 담당 법인)

---

### Step 2. 경험 쥐어짜기 (원자화 무기 매핑)

**이 단계에서 텍스트 작성 절대 금지 — 키워드·방향만 잡을 것**

진행 방법:
1. 원자화 무기 노트 전체 목록 펼치기
2. Step 1에서 발굴한 JD 키워드에 각 무기를 1:1 매핑
3. 매핑 강도 분류: ★★★(직결) / ★★(보조) / ⚠️(억지, 쓰지 말 것)
4. Step 1 조사에서 새로 발견된 키워드로 무기 추가·교체 여부 결정
5. **갭 정리** — 경험 없는 항목 솔직하게 명시, 커버 전략 수립

**Outside-in 원칙**
- JD 먼저, 내 경험은 JD에 끌어다 붙이는 것
- 억지 연결은 면접에서 무조건 깨짐
- 잘 맞는 것 몇 개를 깊게 > 얕게 많이

---

### Step 3. 글 작성

자소서 문항이 있으면 → 문항별 텍스트
포트폴리오라면 → 섹션 방향·키워드 확정 후 작성

글쓰기 세부 규칙은 아래 섹션 참조.

---

### Step 4. 피드백 루프 (다각도 시뮬레이션)

세 입장에서 각각 따로 읽을 것. 동시에 읽으면 관점이 섞임.

| 읽는 사람 | 체크 포인트 |
|---|---|
| **개발자 면접관** | 기술 수치가 실제 구현과 일치하는가 / 왜 그 선택을 했는가 |
| **HR 담당자** | JD 키워드 커버리지 / 수치 과장 여부 / 인재상 반영 여부 |
| **서류 심사자** | 항목 간 일관성 / 같은 수치가 다르게 표기되지 않는가 |

## 2. 자소서·포트폴리오 글쓰기 요령

### 2-1. 문체 원칙

**개조식 필수**
- 명사형 또는 짧은 동사형으로 마무리
- `~차단 / ~0건 / ~달성 / ~격리` 형태
- bullet 길이: 80~100자 수준 (너무 짧으면 맥락 손실, 너무 길면 서술형)

```
✅ - Redis Lua 원자화 → Ghost Reservation 0건
✅ - REQUIRES_NEW 단건 격리 → 배치 오염 차단 → DLQ 격리
❌ - 저는 Redis Lua 스크립트를 원자화하여 문제를 해결했습니다
❌ - 전체 서비스가 응답 불가 상태로 진입하는 현상이 확인되었습니다
```

**역피라미드 필수**
- 각 섹션 첫 줄에 핵심 수치 또는 결과 배치
- 읽는 사람이 임팩트를 먼저 파악하도록

**절대 금지**
- `~이어짐` `~되었습니다` `~했습니다` `~있습니다` 서술형 문미
- 수치 없는 추상 표현: "크게 향상", "효율적으로", "안정적으로"
- 실제 구현 없는 내용 주장

---

### 2-2. 섹션 구조 (포트폴리오용)

```
**문제**
- bullet × 3개

**해결**
- bullet × 3개 (분석이 많은 섹션은 +1 허용)
- 분석(Analyze)은 별도 소제목 없이 해결 bullet 안에 흡수
  예) "- 단순 스케일아웃 제외 — DB 커넥션 풀 소모량이 비례 증가하여 근본 해결 불가"

**결과**
- bullet × 2~3개
```

- 섹션당 A4 반장 이내
- 수치 없는 섹션(인프라, 테스트 등)은 소제목 자유롭게

---

### 2-3. 수치 처리 원칙

| 구분 | 처리 방법 |
|---|---|
| 부하테스트·EXPLAIN ANALYZE 실측값 | 그대로 기재 |
| 로직 기반 추정치 | `~80%` 처럼 `~` 붙이기 |
| 추정 근거 없는 수치 | 사용 금지 |

---

### 2-4. 다이어그램 원칙

- 방향: LR (좌→우) 우선 — 세로로 길게 늘어나는 TD 지양
- BEFORE/AFTER 비교: subgraph 두 개 + 연결 엣지 수평 배치
- **노드 수: 섹션당 5개 이하**
- sequenceDiagram은 단계가 명확할 때만
- **다이어그램 역할**: 텍스트를 대체하는 게 아니라 보조 수단 — 핵심 하나만 집중. 여러 개를 한 그림에 넣으면 산만해짐

**이미지 생성 에이전트에게 명세 줄 때 (mermaid가 아닌 실제 그림 생성 시)**
- **한글 노드 금지**: 에이전트가 한글 텍스트를 그대로 박음. 노드 레이블은 영문으로만
- **코드 용어 직접 표기 금지**: `InputStream`, `MultipartFile` 같은 코드 용어를 명세에 쓰면 텍스트로 박힘. 시각적 표현으로 묘사할 것
  - "파이프가 좁아지는 형태" (병목 표현)
  - "경유지 박스가 있다 vs 없다" (단계 제거)
  - "빨간 실린더" (디스크 I/O), "초록 박스" (메모리)
- **대기업 기술블로그 스타일**: 실제 컴포넌트 이름 박스 + 화살표. 아이콘 은유(창구, 물결)는 초등 교육 자료처럼 보임

---

### 2-5. 이력서 불릿 포맷 (자소서와 구분)

자소서가 아닌 **이력서** 프로젝트 불릿은 별도 포맷을 따른다.

**구조**: 도메인(상황) + 문제 + 해결 + 결과 — 자연스러운 한 문장

**공통 규칙**
- `→ → →` 3단 화살표 금지 — `, 및/으로` 자연스럽게 연결
- 해결 과정 세부사항(가설 기각, 단계별 진행 등)은 제거하고 핵심만

---

#### 이력서 + 포트폴리오 제출 시

역할: 궁금증 유발, 포트폴리오 클릭 유도

- 불릿 길이: **80~100자**
- 해결 과정: 제거 (결과만)
- 결과 수치: **하나만** (가장 임팩트 있는 것)
- 기술 선택 이유: 생략

```
✅ - 부하 테스트 중 내부 executor 병목 확인, VirtualThreadPerTaskExecutor 교체로 RPS 96 → 652 (+580%) 달성
```

---

#### 이력서 단독 제출 시 (포트폴리오 없음)

역할: 그 자체로 판단 근거 — 불릿이 스스로 설명해야 함

- 불릿 길이: **100~130자**
- 해결 과정: **핵심 접근법 1줄 포함** (왜 그 선택인지 힌트)
- 결과 수치: **1~2개 허용**
- 기술 선택 이유: 한 단어라도 힌트 포함 ("분산락 오버헤드 회피", "디스크 거치지 않고 RAM 처리" 등)

```
✅ - 부하 테스트 중 Spring VT가 Tomcat 수신 스레드만 바꾼다는 구조 파악, 내부 executor를 VirtualThreadPerTaskExecutor로 교체해 RPS 96 → 652 (+580%), Avg 1.1s → 249ms 달성
```

---

### 2-6. 프로젝트 소개 3줄 포맷

[핵심기술/아키텍처]로 [도메인 문제]를 처리하는 [성격] 플랫폼.

### 2-6. 완성 후 자가 체크리스트

- [ ] 수치·기술 설명이 실제 코드/구현과 일치하는가?

## 3. 원자화 파일 작성 방식

### 원자화란?

프로젝트 경험을 **주제별 독립 파일**로 쪼개는 것.
하나의 경험 파일 = 하나의 기술 문제·해결 스토리.

목적: JD가 주어지면 태그로 빠르게 필터링 → 자소서/포트폴리오 초안 조합.

---

### 파일 구조

```markdown
---
기술스택: [Java, Kafka, Redis, ...]
키워드: [트랜잭션격리, 배치오염, DLQ, ...]
글자수: 약 900
업데이트: 2026-04-20 코드 팩트 확인 완료
---

# [무기명]

**프로젝트**: 프로젝트명
**수치**: 데이터 유실 0건 / 배치 오염 단건 격리

## 문제
- bullet

## 해결
- bullet

## 결과
- bullet

---
태그: #역량/비동기메시지 #역량/트랜잭션 #수치있음 #핵심무기 #프로젝트/커머스플랫폼
```

---

### 태그 시스템

#### 역량 태그 (JD 키워드 매핑용) **백엔드 개발자 기준 예시임. 직무에 맞게 변동해야함.**

| 태그 | 해당하는 JD 표현 |
|---|---|
| `#역량/Java-Spring` | Java/Spring/JPA 이해도 |
| `#역량/비동기메시지` | 비동기 처리, 메시지 기반 아키텍처, Kafka |
| `#역량/성능최적화` | 성능·확장성, 응답속도 개선, 쿼리 최적화 |
| `#역량/동시성` | 멀티스레딩, 동시성 제어, Race Condition |
| `#역량/분산시스템` | 분산 환경, 정합성, 분산락 |
| `#역량/트랜잭션` | 트랜잭션 설계, 데이터 정합성 |
| `#역량/DB-RDBMS` | MySQL/PostgreSQL, 인덱스, 쿼리 튜닝 |
| `#역량/DB-NoSQL` | Redis, Elasticsearch, MongoDB |
| `#역량/데이터파이프라인` | ETL, 데이터 수집·적재 |
| `#역량/배치` | 배치 처리, 스케줄러, 대용량 처리 |
| `#역량/인프라배포` | CI/CD, K8s, 배포 자동화, 클라우드 |
| `#역량/MSA` | 마이크로서비스, 멀티모듈 |
| `#역량/테스트TDD` | TDD, TestContainers, 단위/통합 테스트 |
| `#역량/AI-LLM` | LLM 에이전트, RAG, Function Calling |
| `#역량/트러블슈팅` | 장애 대응, 병목 진단, 실측 검증 |

#### 강도 태그

| 태그 | 의미 |
|---|---|
| `#핵심무기` | 수치 검증 + 스토리 완성된 최상위 무기 |
| `#수치있음` | 실측 수치 보유 |

#### 노트당 태그 수: 4~7개 적당

---

### 무기 노트 파일명 규칙

```
T{번호}_{기술명}_{주제}.md
예) T2_Kafka_트랜잭션_파이프라인.md
```

경험(소프트) 노트:
```
E{번호}_{경험명}.md
예) E2_기업협력_협업.md
```

---

### 무기 마스터 목록 관리

별도 파일(`무기_마스터_목록.md`)에 전체 무기 인덱스 유지.

```markdown
| # | 무기 | 검증 수치 | 프로젝트 | 비고 |
|---|---|---|---|---|
| T1 | [무기명] | [수치] | [프로젝트명] | [특이사항] |
```

JD가 들어올 때마다 이 목록에서 매핑 후 해당 파일만 꺼내 씀.

---

### 팩트 검증 원칙

원자화 파일에 쓴 수치·코드 동작은 반드시 실제 코드 또는 테스트 결과와 대조 후 frontmatter에 확인 날짜 기록.

```
업데이트: 2026-04-20 코드 팩트 확인 완료 (KafkaConfig.java, PurchaseRepository.java)
```

확인 안 된 수치는 `~` 붙여서 추정임을 표시.

## 4. AI 에이전트 활용 방식

AI와 협업할 때 아래 맥락을 항상 제공할 것.

### 매 작업 전 전달 정보

1. 현재 어느 단계인지 (Step 1~4 중)
2. 회사명 + JD 원문 또는 링크
3. 원자화 무기 목록 (또는 관련 무기 파일)

### 출력 형식 규칙

- **자소서 본문**은 마크다운 인용구(`>`) 또는 코드블록 없이 **플레인텍스트**로 출력
- 이유: 복붙해서 바로 쓸 수 있어야 함. 서식이 섞이면 불편함

### AI 역할별 가이드라인

| 단계 | AI 역할 | 금지 행위 |
|---|---|---|
| Step 1 | 공홈·블로그·다트·합격후기 조사 및 정리 | JD만 보고 무기 매핑 시작 |
| Step 2 | 매핑 테이블 작성, 갭 분석 | 텍스트 초안 작성 |
| Step 3 | 초안 작성, 개조식 교정 | 억지 무기 포함 |
| Step 4 | 각 입장별 시뮬레이션 피드백 | 세 입장 동시에 섞어서 피드백 |

### 수정 시 원칙

- 글자 수 맞추려고 패치식 수정 금지
- 문제가 생기면 **구조 재설계** (문장 하나 끼워넣기는 결국 전체 품질 저하)
