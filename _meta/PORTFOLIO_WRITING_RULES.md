# 포트폴리오 작성 규칙 (AI 작업용 프롬프트)

이 문서는 기술 포트폴리오 작성 시 AI에게 전달하는 작업 규칙이다.
코드베이스를 읽고 포트폴리오를 작성하거나 수정할 때 이 규칙을 반드시 따를 것.

---

## 1. 접근 방식 — Outside-in

1. **JD 먼저** — 공고의 필수/우대 역량을 카테고리별로 순위 매기기
2. **무기 목록** — 프로젝트에서 실제로 구현한 경험을 수치와 함께 정리
3. **매핑** — JD 키워드에 맞는 무기만 골라서 연결 (억지 연결 절대 금지)
4. 같은 무기를 여러 섹션에 써도 되지만 **반드시 다른 각도**로
5. 모든 무기를 다 쓸 필요 없음 — 잘 맞는 것 몇 개를 깊게

---

## 2. 문체 원칙

### ✅ 개조식 (필수)
- 명사형 또는 짧은 동사형으로 마무리
- `~했다 / ~차단 / ~0건 / ~달성` 형태
- bullet 길이: **80~100자 수준** (너무 짧으면 맥락 손실, 너무 길면 서술형)

```
✅ - Redis Lua 원자화 → Ghost Reservation 0건
✅ - REQUIRES_NEW 단건 격리 → 배치 오염 차단 → DLQ 격리
✅ - 피크 트래픽 시 MySQL 커넥션 풀 고갈 → 신규 요청 대기 → 전체 서비스 응답 불가 (k6 실측)
❌ - 저는 Redis Lua 스크립트를 원자화하여 Ghost Reservation을 0건으로 만들었습니다
❌ - 전체 서비스가 응답 불가 상태로 진입하는 현상이 확인되었습니다
```

### ✅ 역피라미드 (필수)
- 각 섹션 **첫 줄에 핵심 수치 또는 결과** 배치
- 읽는 사람이 임팩트를 먼저 파악하도록

```
✅ > **3,000 VU · Peak 2,900 RPS · 에러율 0%**
✅ > **데이터 유실 0건 · 오버부킹 0건 / Entry 200건 = Purchase 200건**
```

### ❌ 절대 금지
- `~이어짐` `~되었습니다` `~했습니다` `~있습니다` 서술형 문미
- 수치 없는 추상 표현: "크게 향상", "효율적으로", "안정적으로" (수치로 대체)
- 실제 구현 없는 내용 주장

---

## 3. 섹션 구조

각 섹션은 반드시 아래 3개 소제목으로 구성:

```
**문제**
- bullet × 3개 (기본)

**해결**
- bullet × 3개 (기본, 분석이 많은 섹션은 +1 허용)
- Analyze(분석)는 별도 소제목 없이 해결 bullet 안에 흡수
  예) "- 단순 스케일아웃 제외 — DB 커넥션 풀 소모량이 비례 증가하여 근본 해결 불가"

**결과**
- bullet × 3개 (기본, 2개도 허용)
```

- **섹션당 A4 반장 이내** — 간결하게
- 수치가 없는 섹션(인프라, 테스트 등)은 예외로 소제목 자유롭게

---

## 4. 수치 처리 원칙

| 구분 | 처리 방법 |
|---|---|
| k6 등 부하테스트 실측값 | 그대로 기재 |
| EXPLAIN ANALYZE 실측값 | 그대로 기재 |
| 추정치 (로직 기반 계산) | `~80%` 처럼 `~` 붙이기 |
| 추정 근거 없는 수치 | 사용 금지 |

---

## 5. 다이어그램 원칙 (A3 출력 기준)

- 방향: **LR (좌→우) 우선** — 세로로 길게 늘어나는 TD 지양
- BEFORE/AFTER 비교 시 subgraph 두 개 + 연결 엣지 추가 (수평 배치 강제)
  ```
  BEFORE -.->|"개선"| AFTER
  ```
- 노드 수: **섹션당 5개 이하**
- sequenceDiagram은 단계가 명확할 때만 사용

---

## 6. JD 키워드 매핑 체크

포트폴리오 완성 후 아래 항목 반드시 체크:

| # | 체크 항목 |
|---|---|
| 1 | 필수/우대 역량이 어느 섹션에서 커버되는가? 빠진 것 없는가? |
| 2 | 수치·기술 설명이 실제 코드/구현과 일치하는가? |
| 3 | 개조식 유지, 서술형 문미 없음, bullet 길이 적정한가? |
| 4 | 다이어그램이 내용과 일치하고 A3 출력에 적합한가? |
| 5 | 처음 보는 사람이 맥락 없이 이해할 수 있는가? 약어 설명 있는가? |
| 6 | 각 섹션 첫 줄에 수치/결과가 있는가? |
| 7 | 섹션별 A4 반장 이내, 너무 얇거나 두꺼운 섹션 없는가? |
| 8 | 실제보다 부풀려진 수치, 경험 없는 내용 주장 없는가? |

## 7. 프로젝트 소개 작성 원칙

맨 앞 프로젝트 소개 3줄은 **JD 핵심 키워드를 첫 문장에** 배치:

```
[핵심기술/아키텍처]로 [도메인 문제]를 처리하는 [성격] 플랫폼.

- 구조: [멀티모듈 구조] — [핵심 통신 방식]
- 규모: [VU/RPS/노드 수 등 수치]
- 협업: [멘토/팀 구성]
```
